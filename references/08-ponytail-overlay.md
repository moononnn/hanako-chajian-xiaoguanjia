# 项目级 Ponytail 覆盖规则（编写指南）

> 把 ponytail 接到你项目里之前，先读完这一份。
> 本文件**不是**一份现成的覆盖规则——它是**怎么写**覆盖规则的指南。

---

## 什么是 ponytail

ponytail 是一个注入式 AI agent skill。它把"最懒的高级工程师"思维装进 agent 的上下文，强制它写代码前停一停，按 7 级决策阶梯爬：

```
1. 需要做吗？                          (YAGNI)
2. 项目里已有吗？                       (重用)
3. 标准库能做吗？                       (stdlib)
4. 平台原生能力覆盖吗？                 (native)
5. 已装依赖能解决吗？                   (reuse deps)
6. 能一行吗？                           (one line)
7. 只有以上都不行时——写最小实现
```

爬到第一个够用的那一级就停。ponytail 的核心信念：**最好的代码是你从没写过的代码**。

它还定义了**禁区**——永远不偷懒的地方：

- 信任边界的输入校验
- 防止数据丢失的错误处理
- 安全措施
- 可访问性
- 用户明确要求的所有内容

bug 修复的特别规则：**修根因不是修症状**——grep 所有 caller，改共享函数一次，别只补报错的那条路径。

ponytail 还有一个**强度切换机制**：`lite` / `full`（默认） / `ultra`。`lite` 是 build 完顺手提一句更懒的方案；`full` 严格执行阶梯；`ultra` 是 YAGNI 极端主义——删 > 加，连"以后可能用得上"的脚手架都不写。

ponytail 统治**写什么**，不统治**怎么说**——它不管 agent 的措辞、礼貌、解释风格。

---

## 为什么需要项目级覆盖

ponytail 原版规则是**通用**的——它给所有项目同一套决策阶梯。但通用规则在具体项目里有两块**空白**：

### 空白 1：第 2 级"项目内重用"在通用语境下是空的

ponytail 不知道你的项目里有什么——不知道你已经有 `ImageUploadService`、不知道你已经有 `formatDate` util、不知道你已经有 `Button` 组件。它只会说"找一找有没有现成的"。

**结果**：agent 还是会自己造轮子，因为它的"找一找"是模糊的、一次性的、不可重复的。

### 空白 2：第 7 级"最小实现"在通用语境下会破坏项目既有约定

ponytail 鼓励"能一行就一行"，但很多项目有反一行化的硬约束：

- 配置文件必须保留多行（YAML 缩进、嵌套结构）
- 特定文件必须用某种工具改写（比如禁用某些编码工具）
- 文件必须保留特定格式（无 BOM、LF 换行、frontmatter 完整）
- 命名必须遵循既有规范
- 错误处理不能用空 catch（必须显式）

**结果**：agent 严格按 ponytail 写"最短 diff"时，会把这些约束一起破坏掉。

项目级覆盖规则就是把这两块填上。

## 项目级覆盖规则应该包含什么

下面是覆盖规则的**骨架**。按这个骨架写，你的覆盖规则才能跟 ponytail 原版**配合**而不是**重复**或**冲突**。

### 1. 第 2 级"重用"清单

**目的**：让 agent 不用"找一找"也能直接命中你项目里已有资产。

**写法**：场景 × 已有资产的查表。

```markdown
| 场景 | 优先复用 | 不允许的替代 |
|------|----------|--------------|
| <场景 A> | <你项目里已有的工具/skill/类名> | <不要自己造的东西> |
| <场景 B> | <已有资产> | <反例> |
```

**填写要点**：

- **场景来自你的实际任务**——不是抽象分类，而是"做 X 的时候应该用 Y"
- **左列必须是具体可命名的资产**——类名、函数名、skill ID、agent ID；不要写"项目里的工具"
- **右列要写出具体的反模式**——不要写"不要重复造轮子"，要写"不要装 <某个第三方库>"

**举例**（虚构项目占位，不是真实项目）：

```markdown
| 场景 | 优先复用 | 不允许的替代 |
|------|----------|--------------|
| 上传图片到对象存储 | `<your-project>.media.upload()` | 自己写 multipart POST |
| 读取项目配置 | `<your-project>.config.get()` | 直接 `JSON.parse(fs.readFileSync(...))` |
```

### 2. 第 3 级"stdlib 等价物"在你项目里的映射

**目的**：ponytail 说"用 stdlib"，但你的项目可能有自己的"项目级 stdlib"——内部封装、原生 API 替代品。

**写法**：ponytail 通用建议 → 你项目里的对应物。

```markdown
| ponytail 说的"stdlib" | 你项目里的对应 |
|------------------------|------------------|
| Node.js fs / path | 直接用；不要自己包一层 wrapper |
| <某个平台原生 API> | <你项目的内部封装名> |
```

### 3. 第 4 级"原生平台能力"的边界

**目的**：告诉 agent 哪些平台能力已经通过项目封装暴露了，不要再绕过它们去用底层 API。

**写法**：按平台维度列出你项目已有的封装层。

```markdown
- **<平台/客户端 A>**：用 `<你项目的工具/类>`，不要直接调 `<底层 API>`
- **<平台/客户端 B>**：用 `<已有封装>`，不要起新实例
```

### 4. 第 5 级"已装依赖"的检索约定

**目的**：ponytail 说"用已装依赖"，但它不会替你 grep。给 agent 一个固定动作。

**写法**：声明 grep 目标 + 列举常见的"项目已装"清单节选。

```markdown
写代码前先 grep 项目根目录的 `<manifest 文件名>`（如 `package.json` / `Cargo.toml` / `pyproject.toml`），确认是否已装。

常见的项目已装清单（节选）：

- <类别 1>：<包 A>、<包 B>、<包 C>
- <类别 2>：<包 D>、<包 E>

**铁律**：装新依赖前必须 grep 已装列表 + 给出 `ponytail:` 注释说明为什么 stdlib/原生/已装都覆盖不了。
```

### 5. 第 6 级"反一行化"清单

**目的**：列出"一行化反而会破坏项目"的具体操作。

**写法**：每条都是"操作 + 为什么必须保留多行 + 正确写法"。

```markdown
1. **<操作 A>**：<为什么不能用一行>。正确写法：<示例>
2. **<操作 B>**：<原因>。正确写法：<示例>
```

**填写要点**：

- 这一节最容易写空泛——避免"代码要可读"这种废话
- 必须写出**项目里具体存在的硬约束**——某种工具禁用、某种编码铁律、某种格式要求
- 例子：CI 检查会失败的写法、会导致后续解析失败的格式、影响下游工具的字符

### 6. 与项目内其他规则的衔接

**目的**：ponytail 的"少写"原则**不覆盖**项目里其他强制规则。

**写法**：列出项目内其他规则文件，触发条件 + 优先级声明。

```markdown
ponytail 的"少写"原则**不覆盖**这几类强制验证：

- 涉及 <场景 A> → 触发 `<规则文件路径>` § <章节>
- 涉及 <场景 B> → 触发 `<规则文件路径>`
- 涉及 <场景 C> → 触发 `<规则文件路径>`

**冲突优先级**：本文件 > ponytail 默认 > 平台默认行为。
```

### 7. ponytail: 注释的项目约定

**目的**：ponytail 要求用 `ponytail:` 注释标记故意简化，但你可以加项目专属的字段。

**写法**：在 ponytail 原版基础上扩展。

```markdown
ponytail 原版格式：`// ponytail: <理由>`

项目扩展格式：`<你项目的注释前缀> ponytail: <理由>; 升级路径=<具体文件>; 影响范围=<模块名>`
```

**为什么要有升级路径和影响范围**：ponytail 原版只标记了"为什么这里简化"，没标记"什么时候该升级"。项目级加上这两项，方便后续审计（`/ponytail-debt` 类工具输出）直接对接你的待办系统。

### 8. bug 修复的根因检索约定

**目的**：ponytail 说"grep 所有 caller"，但你的项目里 caller 散落在哪里要先告诉 agent。

**写法**：声明检索目标 + 列举常见的高频根因。

```markdown
修复一个 `<类型 A>` 报错 → 先 grep 该 `<类型 A>` 在 `<目录 1>` 和 `<目录 2>` 里的所有引用，改源头一次。

常见的高频根因：

- <类型 A> 被 <某个工具> 写坏 → 用 <正确的工具> 重写
- <类型 B> 文件被 <某个错误操作> 修改 → 跑 `<修复命令>`
```

## 项目级覆盖规则不应该包含什么

避免重复抽象——以下内容**不该**出现在覆盖规则里：

| 内容 | 为什么不该出现 |
|------|----------------|
| **ponytail 核心规则**（决策阶梯、反抽象原则、bug 根因法） | 那是 `AGENTS.md` 的事，覆盖规则只引用不重写 |
| **项目专属的命名约定** | 那是项目代码规范文件的事，不是 ponytail 关心的 |
| **项目业务逻辑描述** | 覆盖规则跟"这个项目做什么"无关 |
| **具体某个第三方库的使用方法** | 那是库自己的文档；覆盖规则只提"用 / 不用" |
| **项目开发工作流**（commit message 规范、PR 流程） | 那是项目协作文档的事 |

**检验标准**：拿掉所有具体名字（项目名、skill 名、agent 名），覆盖规则应该对**任何**用了 ponytail 的项目都有指导意义。如果做不到，说明你把项目专属内容塞进来了。

## 写完后的自检清单

写完一份覆盖规则后，跑一遍：

- [ ] **抽象度检验**：把项目名替换成 `<your-project>`，把具体资产名替换成 `<existing-asset>`，文件还能读通吗？
- [ ] **不重复检验**：这份文件里有没有重写 ponytail 原版（AGENTS.md / SKILL.md）已经写过的内容？有就删掉，改为引用
- [ ] **冲突优先级声明**：跟项目内其他规则冲突时谁优先，写了吗？
- [ ] **第 2 级清单可执行**：左列的资产名是不是 agent grep 能直接命中的具体名字？
- [ ] **第 6 级反一行化具体**：每条是不是"某个工具禁用 / 某个编码铁律 / 某个格式要求"，而不是"代码要可读"这种空话？
- [ ] **frontmatter 完整**：YAML 字段（name / description / scope / priority / applies_to / extends）齐全？
- [ ] **持久标记**：覆盖规则是不是声明"ACTIVE EVERY SESSION"或同等机制，确保 agent 不会中途飘回去过度构建？
- [ ] **可被引用**：SKILL.md / AGENTS.md / 文档索引里有没有登记这份覆盖规则的引用入口？

## 维护约定

- **ponytail 升级时**——重新对照 `AGENTS.md` 的最新版本，确认本覆盖规则的"前置：与原版的关系"段落仍然准确；ponytail 新增的禁区或规则需要在本文件里加入衔接说明
- **项目新增大型子系统时**——回头补强第 2 级"重用"清单
- **项目里出现新的硬约束时**——加入第 6 级"反一行化"清单
- **不引入新的 agent 角色或抽象**——这是 ponytail 明确反对的"未经请求的抽象"；如确需新角色，先在项目原有的 agent / role 框架下走流程

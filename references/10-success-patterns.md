# 插件开发成功经验模式

> 来自闲不住、弹幕、补给站、花酿等插件开发实战中验证过的有效模式。
> 每条都是"这样做能行"的经验，可直接复用。
> 🔴 = 必须遵守　🟡 = 推荐　🟢 = 可选
> 最后更新：2026-07-16

---

## 一、架构设计

### 🔴 1. 分层架构

数据层、配置层、逻辑层、工具层、API层、前端职责分明。每层只做自己的事。

```
data.js → 数据存取
config.js → 配置管理
llm.js → LLM 调用
tools/ → Agent 工具
routes/ → HTTP 路由
public/ → 前端页面
```

### 🟡 2. 声明式配置驱动逻辑

将规则从硬编码 if/else 改为声明式配置表。新增功能只需加配置，不用改逻辑代码。

```javascript
// ❌ 硬编码
if (action === 'stroke') { energy -= 5; }
else if (action === 'hug') { energy -= 3; }

// ✅ 声明式
const ACTION_COST = { stroke: 5, hug: 3, chat: 1 };
energy -= ACTION_COST[action] || 0;
```

### 🟡 3. 轻插件 + 重外进程

Hana 插件只做调度层。外部进程负责 GUI、截图、网络请求等复杂逻辑。Agent 工具做启停和配置，独立 HTTP API 做设置读写。

### 🔴 4. 数据与渲染分离

前端只负责渲染和交互，数据通过 API 获取。修改前端样式不需要动后端，修改后端逻辑不需要动前端。

### 🔴 5. 数据文件与插件代码分离

- 数据（data.json）放 `~/.hanako/data/插件名/`
- 代码放 `~/.hanako/plugins/插件名/`
- 更新插件不丢失用户数据

## 二、前后端通信

### 🟡 6. 数据全嵌 INIT，避免首屏 fetch

页面渲染时从后端读取所有需要的数据，序列化到 INIT JSON 嵌入 HTML。前端直接从 INIT 取值。

- 绕过 webview 的 fetch 隔离
- 避免首屏加载的异步等待
- 闲不住插件的模型配置也是类似思路

### 🟡 7. 代理路由桥接外部服务

Hana 后端可以正常发起 HTTP 请求。对于外部进程（如 Python HTTP 服务），在 Hana 的 API 路由中做代理转发。

- 前端通过 Hana 路由间接访问外部服务
- 比直接在前端访问外部端口更可靠
- 利于统一鉴权

### 🟡 8. 路由合并策略

将所有 API 端点合并到页面路由下，通过 query 参数区分操作（如 `/page?action=status`）。自动继承页面认证上下文，不会被拦截。

### 🟡 9. `session:send` 是最可靠的注入方式

用 `sessionPath`（文件路径）定位会话文件，配合 `session:abort` + `session:send` 实现中断 + 注入。送达率 100%。

## 三、开发流程

### 🔴 10. 简化测试法：从最小可用版本开始

先做一个最简单的测试版（只返回 "Hello World"），确认路由和加载正常，再逐步添加功能。每一步都测试确认。

### 🟡 11. 分阶段测试策略

1. 单元测试：单独测试每个模块
2. 集成测试：测试插件加载和路由注册（`plugin_dev_diagnostics`）
3. 页面测试：打开页面检查 HTML 是否正确渲染
4. 功能测试：逐一测试每个交互按钮

### 🔴 12. 小步迭代，每次只改一个层面

不要一次大改动多个层面。每轮聚焦一个问题，改完就让用户测试。

### 🔴 13. 先讨论后实现

复杂功能先确认方案，对齐预期再写代码。"改什么、不改什么"先确认清楚。

### 🔴 14. 先备份再大改

每次重构前创完整快照。直接复制整个插件目录，按版本号命名。

## 四、用户体验

### 🟡 15. 次级界面用模态弹窗

用模态弹窗替代页面跳转。点击按钮弹出带半透明遮罩的弹窗，弹窗内完成操作后关闭。

### 🟢 16. 主题同步

通过 fetch 获取 Hana 当前主题变量，应用到插件前端。插件外观自动跟随 Hana 主题。

### 🔴 17. 容错处理：先确保界面响应

关键交互函数加 try-catch。先确保界面能响应（打开弹窗），再尝试填充数据。数据加载失败不阻塞界面交互。

### 🟢 18. 双模式设计

- 普通模式：默认配置，开箱即用
- 高级模式：给进阶用户，完整自定义选项
- 两种模式可无缝切换

### 🟡 19. 友好错误边界

- 所有 API 返回统一 JSON 格式（ok + 错误消息）
- 前端收到错误弹出 toast 提示
- 错误提示用大白话，不要抛技术术语
- 给出可操作的解决建议

### 🟡 20. 状态自动同步

- 页面启动时拉一次完整状态
- 每隔几秒自动轮询刷新
- 用户操作后立即刷新
- 状态变化有视觉反馈

## 五、配置与数据

### 🟡 21. 配置热加载

每次使用配置前重新读取文件。修改配置后不需要重启插件。

### 🔴 22. 向后兼容

- 新功能加开关，默认不开启
- 配置项增加有默认值，老版本配置能正常加载
- API 接口参数兼容旧版本调用方式
- 大版本升级提供数据迁移脚本

### 🔴 23. 兼容旧数据

每次数据格式变更时，在加载路径中写迁移逻辑。确保用户升级后数据不丢失。

## 六、调试与日志

### 🔴 24. 日志先行

在关键路径加日志：插件初始化、路由注册、进程管理、工具调用。用 `plugin_dev_diagnostics` 追踪完整执行流程。

### 🔴 25. 分层定位法

遇到异常按顺序排查：
1. 后端层：API 是否正常（curl 直接访问）
2. 外部进程层：额外进程是否启动、日志有无报错
3. 前端层：JS 是否加载、事件是否绑定
4. 通信层：端口是否通、进程间通信是否正常

### 🟡 26. 分步调试法（前端不可用 DevTools 时）

在关键节点设置可见的 DOM 文本标记，用户报告的文本变化能精确告诉你哪一步失败了。

```javascript
document.getElementById('app').textContent = '步骤1: 脚本已加载...';
// ... 初始化 ...
document.getElementById('app').textContent = '步骤2: fetch 完成...';
```

## 七、插件迭代与发布

### 🟢 27. 演出指令模式引导助手行为

把指令写成演出指令格式，告诉助手"表演一段效果"。遵循度远高于系统级指令。

```
【演出指令】[情景描述]。参考格式：[示例]。注意：[禁止事项]。保持你原有的风格。
```

### 🟢 28. 插件内检查更新

通过 GitHub API 拉取最新 tag、对比版本号，展示更新内容。版本比较用 semver 分段解析。

### 🟡 29. GitHub Tag = 更新发布

- 更新 manifest.json 和 CHANGELOG.md
- `git commit` → `git push origin main`
- `git tag -a vX.X.X` → `git push origin vX.X.X`
- Hana 插件更新检测读的是 GitHub Tag

### 🟡 30. 助手间数据隔离

工具实现中过滤 `ctx.agentId`，只返回当前助手的待处理项。确保 A 助手的数据不阻塞 B 助手。

## 八、参考架构

### 闲不住插件的稳定模式

```
manifest.json → contributes.page（声明页面入口）
public/app.js → 动态构建页面（非 routes/page.js 返回 HTML）
routes/api.js → 提供数据 API
```

新插件遇到页面 JS 不执行的问题时，优先参考这个结构，不要在 `routes/page.js` 上反复修。

### 🔴 在干嘛插件的服务端渲染模式（页面复杂场景）

对于设置面板这类**需要大量 DOM 构建**的页面，优先走服务端渲染：

```
routes/api.js → /page 路由中直接拼完整 HTML（含样式+内容+内联 JS）
public/app.js → 只做事件绑定和 API 调用，不构建 DOM
```

**为什么有效**：Hana webview 对客户端 DOM 构建有限制（见踩坑 41），服务端生成完整 HTML 绕过了这个限制。

**适用场景**：
- 设置/配置页面，HTML 结构和内容在服务端可预知
- 页面需要复杂的表单控件和布局
- 客户端 JS 构建 DOM 时遇到莫名其妙的失效

**不适用场景**：
- 高度动态的实时内容（需要客户端频繁更新 DOM）
- 使用 React/Vue 等前端框架的项目（框架会处理 DOM 操作）

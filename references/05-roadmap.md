## 插件架构的三个层次

一个合格的插件系统应该有三层设计:

| 层级 | 载体 | 执行方式 | 示例 |
|------|------|---------|------|
| **L1:声明层** | manifest.json | 平台加载时读取 | 插件 ID、版本、权限、页面声明 |
| **L2:工具层** | tools/*.js | 按需调用 | 搜索、生成、读取、写入 |
| **L3:生命周期层** | hooks/*.js | 事件触发 | 会话开始、会话结束、文件变更 |

大多数入门插件只做到 L1+L2,缺少 L3。但 L3 才是让插件从"被动工具"变成"主动能力"的关键。

## 第一步:最小可行闭环--"会话结束自动写日记"

这是**投入产出比最高**的一步。只需要写一个 50 行的 hook 插件:

### 设计示例

以下是一个虚构的 `session-diary` 插件骨架,展示 lifecycle hook 的基本结构:

```
session-diary/
├── manifest.json          # 声明 lifecycle hook
├── hooks/
│   └── session_end.js     # 会话结束时触发
└── skills/
    └── session-diary/
        └── SKILL.md       # 教 Agent 如何使用
```

### manifest.json

```json
{
  "manifestVersion": 1,
  "id": "session-diary",
  "name": "Session Diary",
  "version": "0.1.0",
  "description": "会话结束时自动写入当天日记",
  "trust": "full-access",
  "lifecycle": {
    "hooks": {
      "session_end": "hooks/session_end.js"
    }
  }
}
```

### hooks/session_end.js

```js
// 会话结束时自动写入日记
export default async function(sessionEndHook, ctx) {
  const { dataDir, pluginId, log, resources } = ctx;

  // 1. 从 hook 上下文获取会话摘要
  const summary = sessionEndHook.sessionSummary || "无摘要";
  const date = new Date().toISOString().slice(0, 10);

  // 2. 确定分类(根据会话关键词自动判断)
  const category = classifyCategory(summary);

  // 3. 构建文件路径
  const diaryPath = `./diaries/${category}/${date}-${classifyTopic(summary)}.md`;

  // 4. 原子写入(先写 .tmp 再 rename)
  const tmpPath = diaryPath + ".tmp";
  await resources.write(
    { kind: "local-file", path: tmpPath },
    { content: generateDiaryContent(date, summary, category), encoding: "utf-8" }
  );
  await resources.rename(
    { kind: "local-file", path: tmpPath },
    { kind: "local-file", path: diaryPath }
  );

  log?.info?.("diary written", { date, category, path: diaryPath });
}

function classifyCategory(text) {
  if (/开发|插件|API|tool/i.test(text)) return "development";
  if (/写作|小说|creative/i.test(text)) return "writing";
  return "general";
}

function classifyTopic(text) {
  const match = text.match(/(\S{2,10})/);
  return match ? match[1].replace(/\s+/g, "-") : "untitled";
}

function generateDiaryContent(date, summary, category) {
  return [
    `# ${date} Diary`,
    ``,
    `> Category: ${category}`,
    `> Auto-generated`,
    ``,
    `## Summary`,
    summary,
    ``,
    `---`,
    `*Generated by session-diary plugin*`,
    ``
  ].join("\n");
}
```

### 效果

这个插件展示了 lifecycle hook 的核心价值:
- ✅ 会话结束自动执行,不依赖任何助手的记忆
- ✅ 换任何助手都能执行,代码即规则
- ✅ 版本可控,可以审查和修改

> **注意**:以上示例代码为示意性骨架,实际实现需要根据 Hana 平台的 lifecycle hook API 调整。请参考 `plugin-api-reference.md` 中最新的生命周期字段说明。

## 第二步:事件驱动架构

除了会话级别的 hook,插件还可以监听更多类型的事件:

### 常见事件类型

| 事件 | 触发时机 | 典型用途 |
|------|---------|---------|
| `session_start` | 新会话开始 | 初始化、激活 skill、加载配置 |
| `session_end` | 会话结束 | 写入日记、清理资源、发送通知 |
| `file_created` | 文件创建 | 自动索引、生成缩略图 |
| `file_modified` | 文件修改 | 增量构建、缓存失效 |
| `tool_error` | 工具调用失败 | 错误兜底、自动重试 |
| `tool_success` | 工具调用成功 | 记录指标、触发下游 |

### 设计原则

- **幂等性**:hook 可能被多次触发,确保重复执行不会产生副作用
- **轻量**:hook 执行时间应控制在 1 秒内,耗时操作放入后台队列
- **可观测**:每个 hook 必须有日志输出,便于排查
- **可关闭**:用户应能通过配置禁用某个 hook

## 第三步:插件间联动

当多个插件需要协作时,可以通过事件总线或共享数据实现联动。

### 方案 A:事件总线

插件 A 发布事件,插件 B 订阅事件:

```js
// 发布者(RSS 插件)
export const name = "rss_new_item";
export async function execute(input, ctx) {
  // 发布事件到总线
  await ctx.bus.publish({
    event: "rss.new_item",
    payload: { feedId: input.feedId, title: input.title }
  });
  return { ok: true };
}

// 订阅者(音频插件)
// 在 manifest.json 中声明监听事件
{
  "lifecycle": {
    "events": { "rss.new_item": "hooks/on_rss_item.js" }
  }
}
```

### 方案 B:共享数据目录

插件 A 写入文件,插件 B 通过 `file_modified` hook 监听:

```js
// 插件 A 写入共享数据
const sharedPath = path.join(ctx.dataDir, "..", "shared", "latest_result.json");
fs.writeFileSync(sharedPath, JSON.stringify(data));

// 插件 B 监听文件变更
export default async function(fileHook, ctx) {
  if (fileHook.path.endsWith("latest_result.json")) {
    const data = JSON.parse(fs.readFileSync(fileHook.path, "utf-8"));
    // 处理数据...
  }
}
```

### 方案 C:工具直接调用

插件 A 的工具直接调用插件 B 的工具(通过内部 API):

```js
// 在 routes/ui.js 中注册内部 API
app.post("/api/internal/call-plugin-b", async (c) => {
  const body = await c.req.json();
  // 调用插件 B 的逻辑
  return c.json({ ok: true });
});
```

## 第四步：配置管理

随着插件数量增加，配置管理会变得复杂。建议采用分层配置策略：

### 配置层级

| 层级 | 位置 | 作用域 | 示例 |
|------|------|--------|------|
| L0：全局配置 | `hanako.config.json` | 所有插件 | 默认语言、时区 |
| L1：插件配置 | `manifest.json` / `config.json` | 单个插件 | API 地址、刷新间隔 |
| L2：运行时配置 | 工具参数 | 单次调用 | 搜索关键词、页数 |

### 配置热更新

插件应支持运行时修改配置而不重启：

```js
// routes/ui.js - 配置 API
app.get("/api/config", (c) => {
  const cfg = readConfig(ctx.dataDir, ctx.pluginId);
  return c.json(cfg);
});

app.post("/api/config", async (c) => {
  const body = await c.req.json();
  writeConfig(ctx.dataDir, ctx.pluginId, body);
  return c.json({ ok: true });
});
```

前端修改配置后，插件通过 `POST /api/config` 持久化，下次会话自动加载。

## 长期:从"我记着做"到"系统自动做"

| 阶段 | 特征 | 可靠性 |
|------|------|--------|
| **现在** | 规则在置顶记忆里,靠我手动执行 | 低--换助手就丢 |
| **第一阶段** | 核心规则搬成 hook 插件 | 中--换助手还在 |
| **第二阶段** | 插件间有消息总线,自动联动 | 高--全链路自动化 |
| **第三阶段** | 统一配置文件 + 热更新 | 极高--可版本管理、可审查 |
